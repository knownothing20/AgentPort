# mcp-remote-agent

MCP Server for AI Agent Remote Development

Enable AI Agents to operate remote Linux servers through MCP protocol, seamlessly connecting local development environments with remote servers.

**Analogy**: VS Code Remote SSH is for humans; mcp-remote-agent is for AI.

**v2.5.0**

---

## Core Principle: mcp-remote-agent.json is the Single Config Entry

> **All mutable configurations are centralized in the `variables` section of `mcp-remote-agent.json`.**
> When deploying to a new environment, only modify this one file, then run `node sync.cjs` to automatically sync to all downstream files.

The `variables` section has two groups:

| Group | Variables | Purpose |
|-------|-----------|---------|
| **Client** | `remoteUrl`, `authToken`, `clientId`, `timeoutMs`, `skillDir`, `nodePath` | Local MCP connection to remote daemon |
| **MCP Registration** | `mcpConfigPath`, `mcpServerName` | Target AI tool's MCP config file path and server name |
| **Server** | `serverPort`, `serverBindHost`, `serverWorkspaceRoot`, `serverEnableDashboard`, `serverExecTimeoutMs`, `serverExecMaxConcurrency`, `serverDaemonDir`, `serverAuditLogPath`, `serverAuthTokens`, `serverAdminTokens` | Remote daemon runtime configuration |

`sync.cjs` automatically syncs these variables to:

| Target File | Synced Content |
|-------------|----------------|
| `package.json` | version, name, description |
| `index.js` | version constant, startup log |
| `SKILL.md` | version number |
| `<mcpConfigPath>` | MCP server registration (variable substitution, path determined by mcp-remote-agent.json's mcpConfigPath) |
| `server/.env` | Server runtime config (auto-generated) |
| `test.cjs` | version number |

## Dynamic Connection (v2.3.0+)

Support dynamic switching between multiple remote servers without restarting MCP service.

### Config File: connections.json

```json
{
  "connections": [
    {
      "name": "prod",
      "description": "Production server",
      "url": "http://YOUR_SERVER_IP:3183",
      "authToken": "your-token",
      "clientId": "your-client-id"
    },
    {
      "name": "dev",
      "description": "Dev server",
      "url": "http://YOUR_DEV_SERVER_IP:3183",
      "authToken": "your-token",
      "clientId": "your-client-id"
    }
  ],
  "default": "prod"
}
```

### Usage

1. **View available connections**:
   ```
   remote_connect()
   ```

2. **Switch to a specific connection**:
   ```
   remote_connect(connection="dev")
   ```

3. **All subsequent operations** will automatically use the current connection until switched again.

| `remote_read` | Read remote file | ✅ | ✅ |

## SSH Direct Connection Mode (v2.4.0+)

Connect to remote servers via SSH without deploying daemon process. Ideal for quick development start.

### Config File: connections.json

```json
{
  "connections": [
    {
      "name": "ssh-dev",
      "type": "ssh",
      "description": "Development server (SSH direct)",
      "host": "YOUR_SERVER_IP",
      "port": 22,
      "username": "root",
      "privateKey": "~/.ssh/id_rsa"
    },
    {
      "name": "daemon-prod",
      "type": "daemon",
      "description": "Production server (daemon mode)",
      "url": "http://YOUR_SERVER_IP:3183",
      "authToken": "YOUR_AUTH_TOKEN",
      "clientId": "YOUR_CLIENT_ID"
    }
  ],
  "default": "ssh-dev"
}
```

### SSH Connection Parameters

| Parameter | Description | Required | Default |
|-----------|-------------|----------|---------|
| `type` | Connection type, must be `ssh` | ✅ | - |
| `host` | Server IP or domain | ✅ | - |
| `port` | SSH port | ❌ | 22 |
| `username` | SSH username | ✅ | - |
| `privateKey` | SSH private key path | ❌ | - |
| `password` | SSH password | ❌ | - |
| `passphrase` | Private key passphrase | ❌ | - |

### Usage

1. **Configure SSH connection**:
   ```bash
   cp local/connections.json.example local/connections.json
   # Edit connections.json, add SSH connection config
   ```

2. **Switch to SSH connection**:
   ```
   remote_connect(connection="ssh-dev")
   ```

3. **Check connection status**:
   ```
   remote_health()
   ```

4. **Start development**:
   ```
   remote_bash(command="ls -la")
   remote_read(path="/path/to/file")
   remote_write(path="/path/to/file", content="Hello World")
   ```

### SSH Mode vs Daemon Mode

| Feature | SSH Mode | Daemon Mode |
|---------|----------|-------------|
| **Deployment** | No deployment needed | Requires daemon deployment |
| **Connection** | SSH direct | HTTP API |
| **Async Execution** | ❌ Not supported | ✅ Supported |
| **Config Management** | ❌ Not supported | ✅ Supported |
| **Workspace Isolation** | ❌ None | ✅ Yes |
| **Use Case** | Quick development, temporary use | Long-term development, production |

### Upgrade from SSH to Daemon Mode

1. **Deploy daemon via SSH**:
   ```bash
   # Upload server files
   scp server/* user@server:~/.mcp-remote-agent/daemon/
   
   # Install dependencies on server
   ssh user@server "cd ~/.mcp-remote-agent/daemon && npm install"
   
   # Start daemon
   ssh user@server "cd ~/.mcp-remote-agent/daemon && nohup node server.js > server.log 2>&1 &"
   ```

2. **Update connections.json**:
   ```json
   {
     "name": "daemon-prod",
     "type": "daemon",
     "url": "http://YOUR_SERVER_IP:3183",
     "authToken": "YOUR_AUTH_TOKEN",
     "clientId": "YOUR_CLIENT_ID"
   }
   ```

3. **Switch to daemon mode**:
   ```
   remote_connect(connection="daemon-prod")
   ```

## remote_setup Guided Connection (v2.5.0+)

The `remote_setup` tool guides users through connection setup without any SSH expertise required.

### Usage

#### 1. Test Connection (don't save config)

```
remote_setup(
  host="YOUR_SERVER_IP",
  username="your-username",
  password="your-password"
)
```

Or using key:

```
remote_setup(
  host="YOUR_SERVER_IP",
  username="your-username",
  privateKey="~/.ssh/id_ed25519"
)
```

#### 2. Save Connection Config

Remove `testOnly` parameter to auto-save:

```
remote_setup(
  host="YOUR_SERVER_IP",
  username="your-username",
  password="your-password",
  name="my-server",
  description="Development server"
)
```

#### 3. Auto Deploy Daemon

After SSH connection succeeds, `remote_setup` automatically:
- Checks if daemon is deployed on the server
- If not, uploads server files
- Installs npm dependencies
- Generates authentication config
- Starts daemon service
- Switches to daemon mode

### Parameters

| Parameter | Description | Required | Example |
|-----------|-------------|----------|---------|
| `host` | Server IP or domain | ✅ | `YOUR_SERVER_IP` |
| `username` | SSH username | ✅ | `your-username` |
| `port` | SSH port | ❌ | 22 |
| `password` | Login password | Password or key | - |
| `privateKey` | Private key file path | Password or key | `~/.ssh/id_ed25519` |
| `passphrase` | Private key passphrase (if protected) | ❌ | - |
| `name` | Connection name | ❌ | Auto-generated |
| `description` | Connection description | ❌ | Auto-generated |
| `testOnly` | Test only, don't save | ❌ | `true` |

### User Experience

Users only need to tell the AI:
- Server address
- Username
- Password (or key path)

Everything else is handled automatically by the AI and tools.

## Deployment Steps (From Scratch)

### Step 1: Prepare Remote Linux Server

Ensure the server has Node.js (v18+) and npm installed.

### Step 2: Upload Server Files

Upload files from `server/` directory to the remote server:

```bash
# On remote server
mkdir -p <serverDaemonDir>

# From local scp (execute locally)
scp server/server.js server/mcp-remote-agent-manager.sh server/package.json YOUR_USER@YOUR_SERVER:<serverDaemonDir>/
```

### Step 3: Configure local/mcp-remote-agent.json (Local)

Copy template and fill in real config:

```bash
cp mcp-remote-agent.example.json local/mcp-remote-agent.json
```

Edit `local/mcp-remote-agent.json`, **only need to modify the `variables` section**:

```json
{
  "name": "mcp-remote-agent",
  "version": "2.5.0",
  "variables": {
    "remoteUrl": "http://YOUR_SERVER_IP:3183",
    "authToken": "your-client-token",
    "clientId": "your-client-name",
    "timeoutMs": "120000",
    "skillDir": "absolute-path-to-skill-directory",
    "nodePath": "node",

    "mcpConfigPath": "path-to-your-ai-tool-mcp-config",
    "mcpServerName": "mcp-remote-agent",

    "serverPort": "3183",
    "serverBindHost": "0.0.0.0",
    "serverWorkspaceRoot": "/home/your-user/your-workspace",
    "serverEnableDashboard": "true",
    "serverExecTimeoutMs": "120000",
    "serverExecMaxConcurrency": "2",
    "serverDaemonDir": "/home/your-user/.mcp-remote-agent/daemon",
    "serverAuditLogPath": "/home/your-user/.mcp-remote-agent/daemon/audit.log",
    "serverAuthTokens": "client1=token1,client2=token2",
    "serverAdminTokens": "token1,token2"
  }
}
```

#### mcpConfigPath Common Platform Reference

Different AI tools have different MCP config file paths. Set `mcpConfigPath` to the corresponding path:

| AI Tool | mcpConfigPath (Windows) | mcpConfigPath (macOS/Linux) |
|---------|-------------------------|----------------------------|
| **WorkBuddy** | `C:\Users\<user>\.workbuddy\mcp.json` | `~/.workbuddy/mcp.json` |
| **Claude Desktop** | `C:\Users\<user>\AppData\Roaming\Claude\claude_desktop_config.json` | `~/.config/Claude/claude_desktop_config.json` |
| **Cursor** | `<project>\.cursor\mcp.json` | `<project>/.cursor/mcp.json` |
| **Windsurf** | `C:\Users\<user>\.codeium\windsurf\mcp_config.json` | `~/.codeium/windsurf/mcp_config.json` |
| **Cline (VS Code)** | Write to VS Code `settings.json` `mcp` field | Same |

> 💡 **When deploying to a new AI tool**, only need to change two variables: `skillDir` (point to new installation) and `mcpConfigPath` (point to new tool's config file), then `node sync.cjs` one-click sync.

#### MCP Registration (Generic)

**For other AI tools**, sync.cjs may not auto-write config file (different format/path). Need to manually add the following to the target software's MCP config:

**JSON format** (e.g., WorkBuddy, Cursor):
```json
{
  "mcpServers": {
    "mcp-remote-agent": {
      "command": "node",
      "args": ["<skillDir>/index.js"],
      "env": {
        "MCP_REMOTE_URL": "<remoteUrl>",
        "MCP_REMOTE_AUTH_TOKEN": "<authToken>",
        "MCP_REMOTE_CLIENT_ID": "<clientId>",
        "MCP_REMOTE_TIMEOUT_MS": "<timeoutMs>"
      }
    }
  }
}
```

**TOML format** (e.g., Codex):
```toml
[mcp_servers.mcp-remote-agent]
command = "node"
args = [ "<skillDir>/index.js" ]
env.MCP_REMOTE_URL = "<remoteUrl>"
env.MCP_REMOTE_AUTH_TOKEN = "<authToken>"
env.MCP_REMOTE_CLIENT_ID = "<clientId>"
env.MCP_REMOTE_TIMEOUT_MS = "<timeoutMs>"
```

> ⚠️ Replace `<skillDir>`, `<remoteUrl>`, `<authToken>`, `<clientId>`, `<timeoutMs>` with corresponding values from mcp-remote-agent.json.

> ⚠️ `mcp-remote-agent.json` contains sensitive information, do not commit to Git.

#### Token Format

- **AUTH_TOKENS**: `clientName=secret`, comma-separated. Example: `my-laptop-codex=codex-abc123,my-desktop-workbuddy=workbuddy-xyz789`
- **ADMIN_TOKENS**: Secret list, comma-separated. Admin token can access Dashboard and `/api/config`
- Recommended format: `{hostname}-{software}-{random-suffix}`, e.g., `my-laptop-workbuddy-a1b2c3d4e5f6`

### Step 4: Run Sync

```bash
cd <skillDir>    # The skillDir value from mcp-remote-agent.json, e.g., ~/.workbuddy/skills/mcp-remote-agent

# Install client dependencies
npm install

# Execute sync (auto-update package.json, index.js, mcp.json, server/.env, etc.)
node sync.cjs

# Preview mode (show diff only, don't write files)
node sync.cjs --dry-run

# Check consistency
node sync.cjs --check
```

### Step 5: Deploy Server .env

After sync, `local/server/.env` is auto-generated. Upload to remote server:

```bash
scp local/server/.env YOUR_USER@YOUR_SERVER:<serverDaemonDir>/
```

### Step 6: Start Remote Daemon

```bash
# SSH to remote server
cd <serverDaemonDir>
npm install
nohup bash mcp-remote-agent-manager.sh >> boot.log 2>&1 &
```

Set up autostart (optional):

```bash
(crontab -l 2>/dev/null; echo "@reboot sleep 10 && /bin/bash <serverDaemonDir>/mcp-remote-agent-manager.sh") | crontab -
```

### Step 7: Verify

```bash
# Local test
node test.cjs

# Or local-only check (remote doesn't need to be online)
node test.cjs --local-only
```

In AI conversation, input `remote connection test`, AI will automatically call `remote_health` to check.

## Tool Reference

| Tool | Description | Version |
|------|-------------|---------|
| `remote_health` | Check remote daemon reachability (prefer GET /healthz) | v1.0 |
| `remote_read` | Read remote file (ETag cache, 304 conditional read) | v1.0 |
| `remote_write` | Write remote file (auto CRLF→LF, remove BOM, `expectedEtag` optimistic lock) | v2.1 |
| `remote_stat` | Get remote file metadata (size, modified time, type) | v2.1 |
| `remote_glob` | Search remote files by glob pattern | v1.0 |
| `remote_bash` | Execute remote command (auto base64 encode special chars) | v2.1 |
| `remote_script` | Execute multi-line script (write to temp file, avoid bash escaping) | v2.1 |
| `remote_status` | Comprehensive diagnostics: connection status, latency, cache hit rate, operation stats | v1.2 |
| `remote_batch` | Batch operations, up to 20 read/stat/glob/bash per request | v2.0 |
| `remote_exec_async` | Async execute long-running command, return taskId immediately | v2.0 |
| `remote_task` | Query async command status and output | v2.0 |
| `remote_config` | Read or modify remote daemon config (.env), auto hot reload after modify | v2.2.1 |
| `remote_setup` | Guided connection setup with auto daemon deployment | v2.5.0 |

### Tool Parameter Details

**remote_health**
- No parameters

**remote_read**
- `path` (required): File path, ⚠️ must be within workspace (WORKSPACE_ROOT), otherwise returns Access denied

**remote_write**
- `path` (required): File path, ⚠️ must be within workspace
- `content` (required): UTF-8 text content (auto clean CRLF→LF and UTF-8 BOM)
- `expectedEtag` (optional): Optimistic concurrency lock, prevent overwriting others' changes

**remote_stat**
- `path` (required): File path, ⚠️ must be within workspace, returns size, modified time, is file/directory

**remote_glob**
- `pattern` (required): glob pattern, e.g., `**/*.ts`, `src/**/*.py`
- `cwd` (optional): Search start directory

**remote_bash**
- `command` (required): bash command to execute (auto base64 encode when contains `$` `` ` `` `\` `!` `"` `#` `;` `&` `|` etc.)
- `cwd` (optional): Working directory

**remote_script**
- `content` (required): Script content, written to remote temp file as-is, not parsed by bash -c
- `interpreter` (optional): Interpreter, default `bash`, supports `bash`, `sh`, `python3`, `node` etc.
- `cwd` (optional): Working directory

**remote_batch**
- `operations` (required): Operation array, each item contains `type` (read/stat/glob/bash) and corresponding parameters

**remote_exec_async**
- `command` (required): bash command to execute asynchronously
- `cwd` (optional): Working directory
- Returns `taskId`, use `remote_task` to poll results

**remote_task**
- `taskId` (required): Task ID returned by `remote_exec_async`

**remote_config**
- `action` (required): `read` or `write`
  - `read`: Read remote current config (token auto-masked)
  - `write`: Write new config and trigger hot reload, **no restart needed**
- `config` (required for write): New .env complete content (full file replacement)

> 💡 `remote_config` is suitable for:
> - Dynamically add/delete client tokens
> - Modify workspace directory
> - Adjust timeout/concurrency parameters
> - Auto reload after modify, service uninterrupted

## Usage Rules (AI Must Follow)

### Automatic Health Check

- **Before first operation**, must call `remote_health` to confirm remote service is reachable
- **When continuous operation interval > 5 minutes**, re-check health status
- **Before write operations**, must confirm connection is normal
- When health check fails, don't continue subsequent operations, inform user directly

### Operation Retry

- Network errors (ECONNREFUSED / ETIMEDOUT / ECONNRESET) **auto retry 1 time**
- Wait 2 seconds before retry
- After 2 consecutive failures, stop and notify user
- Non-network errors (parameter error, 404, permission denied, etc.) **no retry**

### Common Scenario Workflows

| Scenario | Recommended Workflow |
|----------|---------------------|
| View remote file | `remote_health` → `remote_read` |
| Edit remote file | `remote_health` → `remote_read` → modify → `remote_write` |
| Execute remote command | `remote_health` → `remote_bash` (simple) or `remote_script` (multi-line) |
| Search remote code | `remote_health` → `remote_glob` |
| View file info | `remote_health` → `remote_stat` |
| Batch read files | `remote_health` → `remote_batch` |
| Long-running task | `remote_health` → `remote_exec_async` → `remote_task` poll |
| Modify remote config | `remote_config read` → modify → `remote_config write` (auto hot reload) |
| Add new client | `remote_config read` → append `newClientName=newToken` to AUTH_TOKENS → `remote_config write` |
| Connection diagnostics | `remote_status` |

### Escaping and Encoding Rules

| Rule | Description |
|------|-------------|
| ✅ Prefer `remote_write` for file writing | No bash, no escaping issues |
| ✅ Use `remote_script` for multi-line scripts | Script written to temp file, completely bypass bash -c parsing |
| ✅ Use `remote_bash` for simple commands with `$` `` ` `` | Auto base64 encode, no manual handling |
| ✅ `remote_write` auto clean CRLF/BOM | Windows→Linux file writing won't leave `\r` |
| ❌ Don't use `remote_bash "cat > file"` for writing | Goes through bash parsing, escaping error-prone |
| ❌ Don't use `remote_bash` to concat multi-line with `$` `` ` `` | Even with base64 fallback, script mode is more reliable |

### Path Restriction Rules

| Rule | Description |
|------|-------------|
| ✅ File operations use in-workspace paths | `remote_read`/`remote_write`/`remote_stat`/`remote_glob` paths must be within `WORKSPACE_ROOT` |
| ✅ Use relative paths for safety | e.g., `package.json`, `src/main.py`, auto-resolve to in-workspace path |
| ❌ Don't file-operate on out-of-workspace paths | `/`, `/tmp/`, `/etc/` etc. will return `Access denied` |
| ✅ `remote_bash` not restricted by this | Command executes in shell, can access any path |
| ⚠️ Use `WORKSPACE_ROOT/.mcp-remote-agent-tmp/` for temp files | `remote_script` fallback auto uses this path |

### Windows Environment Notes

| Issue | Rule |
|-------|------|
| `curl` aliased | Use `curl.exe` not `curl` (PowerShell maps `curl` to `Invoke-WebRequest`) |
| curl hangs | Add `-4` for IPv4: `curl.exe -4 -s URL` |
| CRLF residual | `remote_write` auto cleans CRLF→LF, no manual handling needed |
| BOM residual | `remote_write` auto removes BOM, `mcp-remote-agent.json` reading also strips BOM |
| PowerShell escaping | Don't use `node -e` in PS to pass complex scripts, write `.cjs` file then execute |
| Test script suffix | Use `.cjs` not `.js` (`package.json`'s `"type": "module"` makes `.js` treated as ESM) |

## Troubleshooting

### Service Startup Failure

| Symptom | Cause | Solution |
|---------|-------|----------|
| Port unreachable | Daemon not started | `cd <serverDaemonDir> && nohup bash mcp-remote-agent-manager.sh >> boot.log 2>&1 &` |
| Multiple node server.js processes | manager.sh duplicate start | `pkill -9 -f mcp-remote-agent-manager; pkill -9 -f 'server.js'; sleep 1; cd <serverDaemonDir> && nohup bash mcp-remote-agent-manager.sh >> boot.log 2>&1 &` |
| EADDRINUSE | Port occupied | `lsof -i :3183` find occupying process and kill |
| 401/403 | Token mismatch | Check local `mcp-remote-agent.json` authToken matches remote `.env` AUTH_TOKENS |
| New token not working | dotenv doesn't override existing env vars | manager.sh already auto unsets, confirm using latest; or use `remote_config write` hot reload |

### Client Connection Failure

| Error | Cause | Troubleshoot |
|-------|-------|--------------|
| ECONNREFUSED | Service not started | Check crontab, view `mcp-remote-agent.log` |
| ETIMEDOUT | Network unreachable | `ping` server IP, check firewall |
| 401 / 403 | Token mismatch | Check local `mcp-remote-agent.json` vs remote `.env` |
| Access denied | Path exceeds WORKSPACE_ROOT | Use in-workspace path or relative path |
| ENOENT | File not found | Check path, note `WORKSPACE_ROOT` restriction |
| 500 | Server error | View `mcp-remote-agent.log` |
| curl hangs | PowerShell alias + IPv6 | Use `curl.exe -4` instead of `curl` |

## Security Recommendations

1. **Don't commit real tokens**: `mcp-remote-agent.json` contains sensitive info, ensure excluded in `.gitignore`
2. **Only share template for public release**: Use `mcp-remote-agent.example.json`, no real addresses or tokens
3. **Workspace isolation**: Remote restricts file access scope via `WORKSPACE_ROOT`, prevent unauthorized access
4. **PowerShell script to remote**: Must use base64 encoding (PS parses `$` `$(...)` etc. as variables)
5. **Multi-client isolation**: Each client uses different `CLIENT_ID`, remote audit log traceable
6. **Command execution limits**:
   - Production recommends `ALLOW_BASH_EXEC=false` to disable `remote_bash`
   - Or use `ALLOWED_COMMANDS=git,npm,node,python3` to limit executable commands
   - Script execution (`remote_script`) always restricted by interpreter whitelist
7. **Admin Token security**:
   - Admin token can access Dashboard and config modify, **don't leak**
   - Recommend separate generation from client token, regular rotation
   - Production can set `ENABLE_DASHBOARD=false` to disable web console
